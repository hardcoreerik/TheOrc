=== WHAT THE TRAINING PIT IS ===
TheOrc trains local LoRA/QLoRA adapters for a multi-agent coding swarm. A "Boss"
model decomposes a goal into 2-4 tasks assigned to worker roles (RESEARCHER,
CODER, UIDEVELOPER, TESTER). The Training Pit (a GUI workflow) is THE intended
way to produce these adapters: capture → review → dataset → Forge train → eval →
deploy to Ollama. Hardware baseline: a single RTX 5070 Ti (16 GB), Gemma 4 12B,
QLoRA. Stated north-star: "five-nines" reliability of structured tool-calling /
planning output.

=== CRITICAL RECENT HISTORY (you must weigh this) ===
1. ORC ACADEMY v1 adapter: 900 mostly human-reviewed examples → 99.3% rubric
   pass, eval_loss 0.266. This is the shipping product.
2. ORC ACADEMY v2 adapter: 1,861 examples, 85% UNSCREENED synthetic
   (Cerebras gpt-oss-120b + Codex). It REGRESSED: eval_loss was LOWER (0.2595,
   "better") but real rubric behavior was WORSE (77.8% vs 99.3%). Root cause:
   57-83% of the synthetic training examples showed a TESTER role WRITING files
   (the exact anti-behavior the boss must avoid). The data wasn't bad — it was
   MISROUTED (good for training a Tester WORKER, poison for the Boss).
3. Two design responses are in flight, read them:
   - ContentAwareDatasets_v1.md — make per-example, per-target SUITABILITY a
     queryable SQL property (Migration v7 example_suitability), so the same
     example can train multiple targets and the Pit surfaces datasets by intent.
   - TrainingFlags_Guide.md — research-backed hyperparameter reference; core
     thesis "loss is a proxy, not the objective."
   - rubric_callback.py + train_lora.py — a just-added "rubric-in-the-loop"
     checkpoint selector (pick best-BEHAVING checkpoint, not best-loss).
