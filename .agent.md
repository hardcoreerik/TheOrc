## What This Project Is

A native C# WPF (.NET 10) AI coding IDE for Windows. Think Cursor or Cline but:
- Runs entirely locally (Ollama models, zero subscription)
- Native WPF — 2ms input latency, ~600MB RAM, instant startup
- Trust-first: every file write shows a diff, every shell command needs approval
- Plan → Review → Execute split: agent proposes, user approves

**GitHub:** private repo — do not hardcode local paths or IPs in source files.

## Key Architecture Patterns

### Adding a new tool
1. Create a static method `Task<string> YourTool(Dictionary<string, object?> args, CancellationToken ct)` in the appropriate `Tools\*.cs` file
2. Register it in `MainWindow.xaml.cs` in the `RegisterTools()` method:
   ```csharp
   _registry.Register(new ToolDefinition
   {
       Name        = "your_tool",
       Description = "What it does",
       Parameters  = new() { { "param_name", "string" } },
       Required    = ["param_name"],
       Handler     = YourToolsClass.YourTool
   });
   ```
3. Add it to the allowed list in `ToolRegistry.GetForProfile()` if it belongs in Coding or Full toolset

### The approval gate (trust model)
- `ShellTools.cs` calls `_approvalQueue.RequestApproval(command)` which suspends until user clicks Approve/Reject in the UI
- `FileTools.WriteFile()` calls `_agentPanel.ShowDiff(...)` then waits for approval
- Never bypass approval for shell commands or file writes — this is the core safety feature

### AgentLoop flow
1. `PlanAsync` → model produces text plan → `StripFakeToolBlocks()` removes hallucinated JSON tool blocks → shown in chat
2. User clicks Execute → `ExecuteAsync` → tool-call loop (max 30 steps)
3. Each step: call model → parse tool calls (JSON native OR text-format fallback) → execute → inject result → repeat
4. Refusal guard: if model returns no tool calls and text contains "I'm sorry / I cannot / I am unable" → inject hard nudge message → continue
5. Unknown tool: `ToolRegistry.OnUnknownTool` → `AgentPanel.ShowUnknownToolCard()` → user chooses Auto-translate/Skip/Implement → result injected as tool result

### Model text-format tool calls
`qwen2.5-coder:14b` sometimes outputs tool calls as plain text:
```
{"name": "write_file", "arguments": {"path": "foo.cs", "content": "..."}}
```
`OllamaClient.TryParseTextToolCalls()` detects and parses these. Results are injected as `role: User` messages with `[Tool result: name]\nbody` format (not native tool result format).

### WPF patterns used
- **Styles in `App.xaml`** — never define shared styles in Window.Resources or inline UserControl.Resources (causes forward-reference crashes)
- **`UniformGrid Rows="1"`** as ItemsPanel for tab bars — makes all tabs equal width
- **`HorizontalScrollBarVisibility="Disabled"`** on the ScrollViewer wrapping a UniformGrid — required so UniformGrid knows its bounded width
- **`ObservableCollection<T>`** for all list-bound data
- **`TaskCompletionSource<string>`** to bridge async UI choices (approval, unknown tool) back to the agent loop

## The 7 Core Tools (available to the agent)

| Tool | Purpose |
|---|---|
| `read_file` | Read any file on disk. Args: `path` |
| `write_file` | Write/overwrite a file. Args: `path`, `content`. Shows diff before applying. |
| `list_files` | List directory contents. Args: `directory`, optional `pattern` |
| `run_shell` | Run PowerShell command. Args: `command`. Requires user approval. |
| `grep_code` | Search code with ripgrep. Args: `pattern`, optional `path`, `file_pattern` |
| `get_outline` | Get class/method outline of a file. Args: `path` |
| `run_tests` | Run the test suite. Args: optional `path`. Auto-detects dotnet/pytest/npm. |

**Do NOT hallucinate tools that don't exist** (e.g. `create_project`, `design_form`, `install_package`).
Use `run_shell("dotnet new winforms -n MyApp")` etc. instead.

| File/Pattern | Reason |
|---|---|
| Any hardcoded IP addresses | Ollama host is user-configurable — read from settings |
| Any hardcoded local paths (`F:\Ai\`) | Security — paths differ per machine |

| The `DiffViewer.Approved`/`Rejected` event wire-up | Agent file writes depend on this being async-correct |

## Common Mistakes & Fixes

| Mistake | Fix |
|---|---|
| Using `yield return` inside a `catch` block | Capture error in `string? err = null`, check after the catch |
| `reader.EndOfStream` blocks async streaming | Use `ReadLineAsync()` returning null for EOF |
| Tab items don't auto-resize to equal width | Use `UniformGrid Rows="1"` as ItemsPanel + `HorizontalScrollBarVisibility="Disabled"` on wrapping ScrollViewer |
| Shared style forward-reference crash | Define shared styles in `App.xaml`, not in `Window.Resources` |
| Model outputs fake JSON tool blocks in plan | `StripFakeToolBlocks()` in `AgentLoop.PlanAsync` strips them |
| Model says "I cannot" instead of using tools | `IsRefusal()` + nudge in `AgentLoop.ExecuteAsync` + CRITICAL RULES in system prompt |
| File locked during publish | `Stop-Process -Name "OrchestratorIDE" -Force` first |
